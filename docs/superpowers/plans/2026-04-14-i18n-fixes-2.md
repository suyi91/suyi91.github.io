## Task 4: Callout Titles i18n ❌ REVERTED

**Files:**
- `src/remarkPlugin/remark-asides.js`
- `src/remarkPlugin/remark-collapse.js`

### What Was Implemented (Then Reverted)

The original plan tried to:
1. Replace module-level `import {t}` with dynamic `useTranslations(lang)` per file
2. Add `getLangFromPath(file.history[0])` to detect language from file path
3. Change transformer signature from `(tree)` to `(tree, file)`
4. Remove `defaultLabel` function, use inline `localLabel(v)` with dynamic `t()`

### Why It Was Reverted

**Root cause was misidentified.** The original analysis assumed `file.history[0]` contained the content file path, but debugging revealed:

1. **Debug finding:** `remark-asides` console debug only appeared for `src/pages/about/about.md` — content collection files (`src/content/blog/en/*.md`, `src/content/blog/zh/*.md`) never triggered the transformer debug
2. **Yet callouts work** — blog posts DO have styled `remark-aside` elements in the output HTML, proving the plugins process content files through a different pipeline entry point
3. **EN posts show Chinese titles** — `dist/blog/en/new-features-test/index.html` has "注意" (Chinese) instead of "Note" (English) for `:::note` callouts, confirming the bug exists

### Key Questions Remaining

1. **How does Astro content collections invoke remark plugins?** — Content files go through a separate pipeline that still respects `markdown.remarkPlugins` in `astro.config.mjs`, but `file.history[0]` is NOT the content file path during content collection rendering
2. **Why does `file.history[0]` only show `about.md`?** — The debug file only captured `about.md` path even though content files produce styled callouts. This suggests Astro's content collection pipeline may not expose the source file path via `file.history` in the same way
3. **What is the actual `file` object for content collections?** — `file.history` may be empty/undefined for content collection files, or `file.path` may hold a different value
4. **Why do EN posts get Chinese titles?** — If `getLangFromPath('')` falls back to `'en'`, EN posts should get English titles. But they show Chinese. This means either:
   - The default `t` (module singleton bound to `config.lang`) is `'zh'` in this environment, OR
   - `file.history` contains a zh-path for ALL files (unlikely)

### Next Steps

1. **Investigate `file.history` for content collections** — Add debug to write `Object.keys(file)` and `file.history`, `file.path`, `file.cwd` to a temp file during content collection rendering
2. **Check `config.lang`** — What is `config.lang` at build time? If it's `'zh'`, the module-level `t` would always return Chinese
3. **Check Astro content collection remark pipeline** — How does Astro pass files to `remarkPlugins` for content collections vs. page files?

### Proposed Fix (After Investigation)

Once root cause is confirmed, apply the dynamic approach:

```js
// In both remark-asides.js and remark-collapse.js
import {useTranslations} from '../i18n/utils.ts';

function getLangFromPath(filePath) {
  return filePath.includes('/zh/') ? 'zh' : 'en'
}

export function remarkAsides(options) {
  options = { label: null, ...options };

  const transformer = (tree, file) => {
    // Need to determine how to get file path for content collections
    const path = file.history?.[0] ?? file.path ?? ''
    const lang = getLangFromPath(path)
    const t = useTranslations(lang)
    // ... rest of transformer with localLabel(v) using t()
  }
  return () => transformer
}
```
